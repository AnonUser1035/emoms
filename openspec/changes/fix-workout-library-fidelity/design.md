## Context

`src/timer/workouts/*.ts` is a hand-authored data library imported from a stripped training log (per the header comments in `emomLibrary.ts` and `repLibrary.ts`: HR/times/per-athlete weights removed, "only the primary prescription... remain"). A structural sweep â€” checking every EMOM block's `durationMin` against `intervalSec Ã— stations.length` where the summary states an explicit round count, checking every rep-workout's `targets` against everything its `summary` claims, and checking every `then: [break]` trailing structure against what `expand.ts` actually does with it â€” turned up eight entries where the coded data doesn't match its own summary. Three of these (`ladder14`, `superset5rft`, `pyr5010`) were already self-flagged in their source comments with "(see summary)", suggesting whoever authored them knew the data was incomplete and left a marker rather than resolving it at the time.

## Goals / Non-Goals

**Goals:**
- Make each of the eight flagged entries' data match what its own summary describes.
- Do this without touching `types.ts` or `expand.ts` â€” every fix must be expressible as data, using patterns the library already uses elsewhere (rest-as-rotating-station, repeated `CircuitPart` lists, explicit sequential per-round stations).
- Add a spec capturing the correctness properties this sweep checked for, so a future data-entry mistake of the same shape has something to be validated against.

**Non-Goals:**
- Re-verifying every workout against the *original* (now-stripped) training log â€” that log doesn't exist anymore. Fidelity here means "the coded data matches its own summary text," not "matches David's original notebook." Where the summary's claim can't be resolved without fabricating a number (`amrapClimbSix`'s missing `roundStep` value), the fix corrects the summary instead of guessing.
- Extending the workout model to support programmatic per-round rep increments for EMOM blocks (i.e., an EMOM-side equivalent of `AmrapWorkout.roundStep`). The `climbingGobletSwing`/`climbingKb` fix works around this by writing out explicit per-round stations instead, consistent with the "data-only" changes this library is meant to support.

## Decisions

- **`kbThen12Min`: bake rest into the rotation, don't switch to `mode: 'interval'`.** An `IntervalWorkout` genuinely models "N rounds of work then rest" (`expandInterval` in `expand.ts` inserts a break after every round), but it has no `then`/chaining concept to flow into the following 12-minute EMOM without splitting into two separate workout entries. Adding a second rotating `rest(120)` station to the existing EMOM block reproduces the same work/rest/work/rest/... timeline as an interval workout would, entirely within the existing `Block` shape (`stationSeconds()` already special-cases `hold`-measure stations to use their own intrinsic duration regardless of the block's default `intervalSec`), and keeps it one workout the athlete picks once. `durationMin` moves from 15 to 25 (5 Ã— (180s work + 120s rest) = 1500s), and the block's trailing `then: [break]` is removed since the rest is now part of the rotation itself.
- **`climbingGobletSwing`/`climbingKb`: unroll the climb into explicit stations, don't add a `roundStep`-like field to `Block`/`Station`.** Adding programmatic per-round increments to the EMOM model would touch `types.ts` and `expand.ts` and affect every consumer of `Block`/`Station`, for the benefit of two entries. Writing out the literal per-round station sequence (a small local generator, the same idiom as `cycle()` in `emom.ts` or `rowRung()`/`highPullRound()` in `repLibrary.ts`) gets the correct runtime behavior today with zero model risk. Total station count = rungs Ã— movements Ã— repeats-through-the-climb (e.g. `climbingGobletSwing`: 4 rungs Ã— 3 movements Ã— 3 times through = 36 stations exactly filling the existing 36-minute `durationMin`), so the rotation modulo in `expandBlock` never wraps mid-sequence.
- **`ladder14`/`pyr5010`: hand-interleave the extra fixed-count target, don't extend the `ladder()` builder.** Both need a fixed-rep target (Russian twists; a second, differently-noted round of pushups) inserted after every rung of an otherwise-climbing ladder â€” something the generic `ladder(scheme, movements)` helper doesn't support (it applies the same climbing count to every movement in the list). Building the target array with a direct `scheme.flatMap(...)` for just these two entries is simpler and safer than generalizing `builders.ts` for a two-entry need.
- **`superset5rft`: represent the 4-minute bear crawl via `notes`, not a new duration-based `Measure` on `RepTarget`.** `RepTarget.count` is always a rep count; there's no time-based variant (unlike EMOM's `hold` measure). A finisher named for its duration rather than a rep count is rare enough in this library (one entry) that adding a model variant isn't justified â€” `{ movement: 'Bear crawl', count: 1, notes: '4 minutes' }` communicates the prescription correctly to the athlete without a type change.
- **`kbComplexAlternating`: repeat the `CircuitPart` list literally (3Ã— and 2Ã—), don't add a "repeat count" field to `CircuitStation`.** The circuit checklist is meant to be a tappable, literal list of what to do; repeating the three parts in place is the direct fix and matches how every other circuit in the library is authored (an explicit ordered list, no implicit repeat semantics).
- **`amrapClimbSix`: correct the summary, don't invent a `roundStep`.** The stripped log has no record of what the real per-round increment was. Any specific number chosen now would be fabricated and silently wrong. The proposal rewrites the summary to describe the workout as it actually runs (flat reps every round) rather than assert a climb that isn't there.

## Migration Plan

Pure data changes in three files (`emomLibrary.ts`, `repLibrary.ts`, `amrap.ts`) plus a new spec capability and expanded tests. No runtime/model changes, no server/Worker involvement, ships in a normal deploy. Rollback is a plain revert of the data edits.
